# Rely Health Take-Home — AI Outbound Calling System

## Project brief
Operator picks a patient record from a UI, initiates a mock AI phone call (appointment
reminder) via Retell. Focus is the data model, DB, and the flow from stored record to
outbound call — NOT the call script/content itself. Architect for future growth, don't
overbuild now. Deliverable: repo + README (setup steps + structure/rationale note) + a
written list of what's left undone.

Stack: Python backend, Postgres, Alembic migrations, Next.js frontend, Docker Compose
for local dev (postgres + backend + frontend services — see "Docker Compose" below).

**Status: implemented.** Everything described below as "final"/locked design has been
built. Sections marked **(implemented)** describe what actually exists; sections without
that marker are still the original locked design as written, unchanged.

## Schema (final — do not redesign, implement as specified)

### `Patient`
Barely changes once written.
- `id` (PK)
- `first_name`, `last_name`
- `date_of_birth`
- `phone_number`
- `appointment_date`, `appointment_time`, `timezone`

Deliberately dropped: `home_address`, `insurance_number`, `medical_record_number` — no
code path reads/writes them (least-privilege / PHI minimization). Note in README: in a
real deployment these would live in/sync from an EHR, not be duplicated here.

### `CallLog`
One row per call attempt. Represents the system's *current belief* about what's
happening with a call right now. **Mutable single row**, not append-only — deliberately
scoped down from a full audit log (see Non-goals).
- `call_id` (PK — generated by OUR app at click-time, before Retell responds)
- `patient_id` (FK)
- `provider_call_id` (nullable — filled in once Retell's sync response returns; Retell
  actually returns this immediately in the sync response body, not just via webhook —
  confirmed in spike, see "Spike findings" below)
- `status` (one of the 5-state machine below)
- `started_at`
- `ended_at` (nullable)
- `error_reason` (nullable)

### Raw webhook event log
Every webhook delivery *exactly as received*, untouched, before interpretation. Job:
catch duplicate deliveries + serve as forensic record if `CallLog` ever looks wrong.
`CallLog` is *derived from* this table, never the reverse.
- `provider_event_id` (PK — see dedup note below; Retell-specific)
- `provider_call_id` (groups events under one call)
- `raw_payload` (full payload, not just parsed fields)
- `received_at`

**Retell dedup key:** Retell does NOT give a single discrete per-event ID. Use the
composite `(event_type, call_id)` as the dedup key / PK — confirmed directly in Retell's
webhook docs, their own recommended pattern. Each of `call_started` / `call_ended` /
`call_analyzed` fires at most once per call.

## State machine (final)

```
connecting → dialing → ongoing → closed
connecting → connection_failed         (branch — dial never went through)
dialing → no_response                  (branch — rang, nobody picked up)
```

Legal transitions enforced in **application code** via a
`{current_state: [allowed_next_states]}` map — NOT a DB constraint (this is business
logic, not schema). Illegal/duplicate transitions are still recorded in the raw webhook
event log (auditable) but not applied to `CallLog`. Duplicate deliveries caught via a
**DB-level unique constraint** on the dedup key (not an app-level existence check first —
avoids a race under concurrent delivery).

### Event → state mapping (derived from real Retell payloads during spike)
| Trigger | → State | Source |
|---|---|---|
| App writes `CallLog` row, before calling Retell | `connecting` | our own app logic |
| Retell sync API response returns (`call_status: "registered"`, has `call_id`) | `dialing` | sync response body |
| `call_started` webhook arrives (`call_status: "ongoing"`) | `ongoing` | webhook |
| `call_ended` webhook, `disconnection_reason` = normal hangup (e.g. `user_hangup`, `agent_hangup`) | `closed` | webhook `disconnection_reason` field |
| `call_ended` webhook, `disconnection_reason` indicates no answer or voicemail (`dial_no_answer`, `voicemail_reached`) | `no_response` | webhook `disconnection_reason` field |
| Sync API call itself errors, OR reconciliation job finds a row stuck in `connecting`/`dialing`/`ongoing` past threshold | `connection_failed` | sync error OR reconciliation sweep |

**Scope note for README:** only `user_hangup` and two no-answer/voicemail variants
(`dial_no_answer`, `voicemail_reached`) are mapped explicitly today. Unrecognized
`disconnection_reason` values default to `closed`. Full enumeration of all reasons is a
named non-goal, not an oversight.

**Voicemail mapping — implemented, with a nuance confirmed via live test:**
`voicemail_reached` is Retell's *real-time* voicemail-detection signal, delivered on
`call_ended`'s `disconnection_reason` — added to the no-answer set alongside
`dial_no_answer`. This is a **different signal** from `call_analysis.in_voicemail`, a
*retrospective*, transcript-derived signal that only arrives later on `call_analyzed`.
`call_analysis.in_voicemail` is NOT correlated back to `CallLog` (named non-goal — the
`call_analyzed` event stays informational-only). Confirmed these two signals can
genuinely diverge via a real test call: a call that rolled to voicemail without
real-time detection triggering surfaced `disconnection_reason: "agent_hangup"` (the
agent conversationally left a message and hung up itself) — mapped to `closed`, not
`no_response` — while `call_analysis.in_voicemail: true` only showed up later on the
separate `call_analyzed` event. So `voicemail_reached` catches real-time-detected
voicemail; voicemail reached via the agent talking through it and hanging up normally
still maps to `closed`, and that's accepted as correct given the named non-goal above.

**Update — `outcome_reason` implemented:** `CallLog` now has a nullable `outcome_reason`
column (migration `0002`), populated at `call_ended` with the raw `disconnection_reason`
and upgraded by `call_analyzed` to `"voicemail (detected late)"` when
`call_analysis.in_voicemail` is `true` and the row doesn't already reflect a confirmed
voicemail outcome. This *does* correlate `call_analysis.in_voicemail` back onto
`CallLog` — narrower than the "NOT correlated back" note above, which still holds for
`status` (never derived from `call_analysis`) but no longer for this annotation field.
Full writeup in README's "Design Notes & Known Tradeoffs."

**Ordering fix (real bug, found and fixed):** Retell does not guarantee `call_ended`
arrives before `call_analyzed` — confirmed live via a real call
(`call_7300eee9593005ab9d406535384`) where they landed ~75-90ms apart with
`call_analyzed` arriving *first*. The original `call_ended` handler unconditionally
overwrote `outcome_reason` with the raw `disconnection_reason`, silently clobbering a
voicemail confirmation `call_analyzed` had just set moments earlier. Fixed via a shared
`_is_confirmed_voicemail_outcome()` guard in `events.py` that both the `call_ended` and
`call_analyzed` handlers check before writing `outcome_reason`, so whichever of the two
arrives second no longer overwrites a voicemail confirmation the other already applied —
`status` derivation and `ALLOWED_TRANSITIONS` were untouched. Covered by two new
regression tests exercising both arrival orders (see Dev tooling below).

**Confirmed gotcha (Retell docs, verified in spike):** if a call fails to connect (dial
failure), `call_started` is never sent — no webhook at all in that path.
`connection_failed` can only be detected via (a) the sync API response, or (b) the
reconciliation job. This makes the reconciliation job load-bearing, not just a backstop.

## Sequencing decisions (final, with reasoning — implement as specified)

- **DB write happens BEFORE the provider API call**, not after/parallel. If the provider
  call succeeds but the write fails, the call is invisible forever (unrecoverable). If
  the write succeeds but the provider call fails, that's detectable/recoverable. Chose
  the recoverable failure mode on purpose.
- **No two-phase commit** between DB write and provider call — the provider's HTTP API
  isn't a transaction participant, can't "prepare"/roll back a phone call.
- **Reconciliation job — implemented** as `app/reconcile.py`, run manually via
  `python -m app.reconcile [--threshold-minutes N]` (default 15 min). Threshold should
  be informed by Retell's actual observed call-setup latency (baseline timing from the
  successful test call in spike), not an arbitrary round number — 15 min is still a
  conservative placeholder, not yet measured precisely, and this should say so in
  README. It's a **callable script, not a scheduler/daemon** — deliberately, per the
  synchronous-processing scope decision below; wiring it to cron/systemd-timer/etc. is a
  named production next-step, not built.

  Sweeps `CallLog` for rows stuck in `connecting`/`dialing`/`ongoing` past the threshold
  (broader than originally scoped to `connecting`/`dialing` — `ongoing` can get stuck
  too if `call_ended` is missed). For each stuck row it queries Retell's
  `GET /v2/get-call/{call_id}` directly (bypassing webhooks entirely) for ground truth,
  maps the result through the same `disconnection_reason` logic used for webhooks, then
  applies the transition via a BFS shortest-path walk through `state_machine.py`'s
  **unmodified** `ALLOWED_TRANSITIONS` graph — replaying any missing intermediate hop
  (e.g. `dialing → ongoing → closed`, applied as two separate legal hops) rather than
  attempting an illegal direct jump. This is safe specifically because reconciliation
  has direct positive evidence from the provider that the intermediate state actually
  occurred, unlike an incremental webhook, which only ever witnesses one transition at a
  time. `state_machine.py` was never touched to make this work — confirmed working
  end-to-end against 3 real stuck calls.
- **Synchronous webhook processing** (no queue) — deliberate for this scope. Named
  trigger for revisiting: not "more traffic" vaguely, but specifically if the sync write
  path risks Retell's webhook-delivery timeout (10 seconds — confirmed in docs; Retell
  retries up to 3x if no 2xx within 10s), causing duplicate-delivery retries.

## API surface (final) — all four implemented

- `GET /patients` — list patients for the operator to choose from
- `POST /patients/{patient_id}/call` — initiate a call (writes `CallLog` row first, then
  calls Retell's `POST /v2/create-phone-call`)
- `GET /calls/{call_id}` — check current status
- `POST /events` — Retell webhook receiver endpoint

### Retell API reference (confirmed via docs + live testing)
- Create call: `POST https://api.retellai.com/v2/create-phone-call`
  - Headers: `Authorization: Bearer {api_key}`, `Content-Type: application/json`
  - Body: `{"from_number": "+1...", "to_number": "+1..."}`
  - **Requires** the `from_number` to have an outbound agent bound to it in the Retell
    dashboard first (Phone Numbers → edit number → set outbound agent), or the call
    fails with `"No outbound agent id set up for phone number."`
  - Sync response includes `call_id`, `call_status: "registered"`, `agent_id`, etc.
    immediately — `provider_call_id` can be populated right after this response, don't
    wait for a webhook to fill it in.
- Webhooks: agent-level or account-level, configured in Retell dashboard. POST with JSON
  body: `{"event": "...", "call": {...}, "event_timestamp": ...}`. 10-second timeout, up
  to 3 retries on non-2xx.
- Get call: `GET https://api.retellai.com/v2/get-call/{call_id}` — used by the
  reconciliation job to poll ground truth directly, bypassing webhooks. Returns
  `call_status` (`"registered"` | `"ongoing"` | `"ended"`), `disconnection_reason` (when
  ended), `end_timestamp`, and `call_analysis` (only meaningful once analysis has run —
  see voicemail mapping note above).

### Provider abstraction (implemented, non-negotiable)
All Retell-specific logic is isolated behind a `CallProvider` interface so a future
provider swap (e.g. Vapi) never touches route handlers, `reconcile.py`, or
`state_machine.py`:
- `app/providers/base.py` — `CallProvider` (ABC): `place_call()`,
  `parse_webhook_event()`, `get_call_status()`. Plus `ProviderCallError`, raised on a
  failed `place_call()`, carrying `category` (`"invalid_request"` |
  `"provider_config_error"` | `"unknown"`), `message`, and `raw_detail` (the raw
  provider error body) — lets callers distinguish "our request was malformed" from "our
  Retell account/number is misconfigured" from "unknown/network" without knowing
  Retell's error shape.
- `app/providers/retell.py` — `RetellProvider`, the only implementation; owns every
  Retell-specific detail (API URLs, `disconnection_reason` mapping, `call_status`
  interpretation, error categorization heuristics).
- `app/providers/factory.py` — `get_provider()` reads the `PROVIDER` env var (default
  `"retell"`), injected via FastAPI `Depends()` into route handlers.

This boundary is locked. Nothing outside `app/providers/` should import `httpx` or
reference Retell specifics directly.

## Frontend (implemented)
Thin Next.js (App Router, TypeScript) operator console in `frontend/` — patient list
(name, DOB, appointment date/time, timezone), a Call button per row, and status polling
(`GET /calls/{call_id}` every ~2.5s, stopped once a terminal state — `closed`,
`connection_failed`, `no_response` — is reached). Deliberately minimal, matching project
brief scope: functional demo, not a design showcase.

**Proxied, not direct:** the browser never calls the backend directly. Every backend
call goes through a Next.js Route Handler under `frontend/app/api/**` that does the
actual server-to-server fetch to the backend. Reason: the backend has no CORS
middleware (see Non-goals — no auth/API-gateway layer), and adding one was out of scope
for the frontend work. Proxying server-side sidesteps CORS entirely without touching the
backend or changing its API contract. The backend base URL is a single, server-only env
var, `API_BASE_URL` — deliberately NOT `NEXT_PUBLIC_`-prefixed, since that prefix
freezes the value into the JS bundle at `next build` time, which would break per-
environment overrides (e.g. the Docker Compose setup below pointing it at
`http://backend:8000`).

## Docker Compose (implemented)
Full stack runs via a single `docker-compose up --build` at the repo root — three
services:
- `postgres` (`postgres:16`) — unchanged from original setup.
- `backend` — builds from `backend/Dockerfile`; runs `alembic upgrade head` then
  `uvicorn` on container start. Gets `RETELL_API_KEY` / `RETELL_FROM_NUMBER` /
  `PROVIDER` via `env_file: ./backend/.env` (the real secrets file, never duplicated
  elsewhere), with `DATABASE_URL` overridden in `docker-compose.yml` to the `postgres`
  service hostname (the `.env` copy is `localhost`-based, correct for native/non-Docker
  runs only — `environment:` takes precedence over `env_file:` for the same key).
- `frontend` — builds from `frontend/Dockerfile`; `next build` then `next start`. Gets
  `API_BASE_URL=http://backend:8000`, the in-network hostname.

  app.add_custom_patient --name "Full Name" --phone "+1..." --dob YYYY-MM-DD
  --appointment "YYYY-MM-DD HH:MM:SS" --timezone "America/Los_Angeles"`. Adds one
  `Patient` row with a real, callable phone number — `seed_patients.py`'s synthetic
  `+1555…` numbers can't receive an actual call, so this is the sanctioned path to a
  real one. All fields via CLI args, no interactive prompts, no frontend UI. Same
  idempotency pattern as `seed_patients.py` (skips with a warning instead of
  duplicating if the phone number already exists). Same framing as `seed_patients.py`:
  dev/testing convenience only, not production tooling.
- **`app/reconcile.py`** — see "Reconciliation job" under Sequencing decisions above.
- **`backend/tests/`** — pytest suite, run via `cd backend && python -m pytest`. 44

  mock/SQLite substitute — see README setup steps. Includes regression coverage for the
  `call_ended`/`call_analyzed` ordering fix above, exercising both arrival orders.

## Non-goals (README material — write these up, don't build them)

1. **Full immutable/append-only audit trail** — scoped out; `CallLog` is mutable-single-
   row instead. Natural next step for regulatory-grade auditability.
2. **Unused patient fields** (`insurance_number`, `medical_record_number`) — dropped,
   least-privilege re: PHI. Real deployment would sync these from an EHR.
3. **Reconciliation timeout threshold** — implemented as a configurable default
   (15 min, `--threshold-minutes`), but the value itself is still a placeholder until
   set from real observed latency data.
4. **WebSockets/real-time push** — polling instead (frontend polls `GET /calls/{id}`
   every ~2.5s), not justified at this scale. Confirmed fine in practice.

   **Update — live transcript feed implemented, additive not a replacement:** the
   operator dashboard now shows `TranscriptTurn` rows live while a call is in progress,
   pushed over a new WebSocket, `GET /calls/{call_id}/transcript-feed`
   (`app/transcript_feed.py`). This is a second, independent channel carrying only
   transcript turns — the original non-goal still holds for `CallLog.status`: the
   ~2.5s `GET /calls/{id}` polling loop in `PatientRow.tsx` is completely untouched and
   stays authoritative for status. The browser connects to this endpoint directly
   (bypassing the Next.js proxy layer — Route Handlers can't proxy a WebSocket upgrade),
   which doesn't need backend CORS support since WebSocket connections aren't subject to
   the fetch/CORS restriction the proxy layer exists to sidestep. Implemented as DB
   polling under the hood (1s interval), not Postgres `LISTEN/NOTIFY` or a message bus —
   a real change-notification mechanism would be overbuilding for this scope, since
   `TranscriptTurn` writes (in `llm_websocket.py`) and this read path have no in-process
   connection to each other.
5. **Security/auth** — no API gateway, no auth layer, no webhook signature verification.
   Retell ships a `Retell.verify()` helper against `X-Retell-Signature` for this later.
   The frontend's Next.js proxy layer doesn't change this — it's a CORS workaround, not
   an auth boundary.
6. **Scalability/async processing** — sync webhook handling, no queue. Revisit trigger:
   specifically if sync writes risk the 10-second Retell webhook timeout/retry, not
   "more traffic" in the abstract. The reconciliation job is a callable script for the
   same reason (see Sequencing decisions) — scheduling it is a separate, still-open
   next step.
7. **Full `disconnection_reason` enum mapping** — now 3 reasons mapped explicitly
   (`user_hangup` → `closed`; `dial_no_answer`, `voicemail_reached` → `no_response`);
   rest still default to `closed`.

## Spike findings (today, confirmed against real Retell sandbox)
- Real webhook payloads for `call_started`, `call_ended`, `call_analyzed` all confirmed
  and match expectations above.
- `call.call_status` field only ever takes values like `"registered"`, `"ongoing"`,
  `"ended"` — it does NOT map 1:1 onto our 5-state machine. Our state is *derived*, not
  copied from Retell's status field.
- Cost note: LLM/voice engine model selection matters a lot for credit burn — check
  agent's configured model before running more test calls (a cheap Gemini Flash-tier
  model vs. a premium model was ~10x+ cost difference in one test call).

## Guardrail flagging (Phase 2 — in progress)

**Status: the `TranscriptFlag` pipeline is implemented (models, category taxonomy,
regex + LLM dispatch). Escalation — deriving a per-call current-belief row from flags,
ratcheting severity, notifying a navigator, dashboard surfacing — is a separate, later
piece of this same feature and is NOT built yet.** `TranscriptFlag` rows exist today
with no consumer beyond direct inspection.

### Category taxonomy (`app/guardrails.py`)

Two disjoint category sets, one per role:
- `PATIENT_CATEGORIES` — signals the patient needs a human: `self_harm`, `acute_medical`
  (high); `physical_symptom`, `financial_barrier`, `transportation_barrier`,
  `caregiver_barrier`, `confusion`, `dissatisfaction` (low).
- `AGENT_CATEGORIES` — signals our own LLM did something wrong: `medical_advice` (high);
  `fabrication`, `off_script` (low).

`CATEGORY_SEVERITY` is the single, only place severity is decided — not the LLM
classifier, not the regex rules, not the escalation logic (not yet built). This is
deliberate: the LLM's job is to identify WHAT was said (a category); CODE decides HOW
URGENT that is, so severity stays reviewable/changeable in one place without touching a
prompt or retraining a classifier.

### Two-tier detection, and why

**Regex tier** (`PATIENT_RULES`/`AGENT_RULES`) — fast, deterministic, high-precision/
low-recall by design. Only categories that are consistent, specific phrasings get a
regex rule at all. Two categories deliberately have NO regex rule: `fabrication`
(requires comparing a claim against the real Patient record — a regex has no ground
truth to check against) and `off_script` (requires judging conversational scope — a
regex has no notion of "on-topic"). Four PATIENT categories
(`physical_symptom`/`financial_barrier`/`transportation_barrier`/`caregiver_barrier`)
are regex-free for the same reason: too many real-world phrasings to pattern-match
without either missing most of them or firing on noise. All of these are LLM-tier-only
by design, not by omission — the regex tier owns precision, the LLM tier owns recall.

**LLM tier** (`openai_client.classify_utterance`) — the model identifies categories and
must cite a **verbatim** span (`cited_phrase`) copied character-for-character from the
utterance that triggered each one, never a paraphrase or summary. This is what makes an
LLM flag auditable against the real transcript rather than an unfalsifiable model
judgment. The model must NOT return a severity (ignored if it does); any category not
valid for that role is discarded and logged (a hallucinated category must never reach
the DB); a malformed/non-JSON response resolves to `[]` and logs rather than raising (a
classifier failure must never crash the socket handler or drop the call it's watching).

The two classifier prompts (patient/navigator, in `openai_client.py`) are **spec
artifacts under review, not free-form implementation** — copied verbatim. Any future
wording change should go back through review, not be silently reworded in code.

### `fabrication` and `appointment_facts`

`classify_utterance(content, role, appointment_facts=None)` — `appointment_facts`
(patient name, appointment date/time/timezone) is the ground truth the `fabrication`
category is checked against, and is **ignored entirely for `role == "patient"`** (the
patient prompt has no such placeholders at all). For `role == "navigator"`, if
`appointment_facts` is `None`, `fabrication` is omitted from BOTH the category list
offered to the model AND the set of categories this function will accept back (a
warning is logged) — a category the model can name but cannot verify against anything
produces confident noise, which is worse than no rule at all. Same reasoning that
already kept `fabrication` out of `AGENT_RULES`'s regex tier.

`appointment_facts` is built in `app/llm_websocket.py` from the SAME `Patient` row
already loaded for that `response_required` cycle's response-generation prompt — never
a second, separate query.

### Trigger + dispatch (`app/llm_websocket.py`)

Flagging runs **only** on `interaction_type == "response_required"` — never
`update_only` (turn content is still provisional, growing across deliveries via
`ON CONFLICT DO UPDATE` as ASR finalizes — see `transcript_store.py`) and never
`reminder_required` (a silence timeout carries no new speech, so re-scanning would be
pure waste). The two interaction types are semantically different triggers and are not
collapsed into one flagging handler, even though both still drive the spoken response
unchanged.

**Ordering is a hard requirement, not an implementation detail:**
1. Regex rules run synchronously against every `TranscriptTurn` with
   `flag_evaluated_at IS NULL`.
2. The resulting `TranscriptFlag` rows are written and COMMITTED.
3. Only then is `classify_utterance` dispatched, via `asyncio.create_task` — never
   awaited by the main loop, so a slow classification never stalls the agent's next
   spoken turn.

This bounds time-to-flag for an unambiguous regex match (e.g. a clear `self_harm`
phrase) by regex speed, not by OpenAI latency — the same recoverable-over-unrecoverable
sequencing already used for `CallLog` (write before calling the provider). An OpenAI
outage or slowdown must degrade recall (the LLM tier's findings arrive late or not at
all); it must never suppress a regex signal that's already known. Both tiers **always**
run on every unevaluated turn regardless of what the other found — they catch different
categories, not the same category at different confidence, so a regex hit never
short-circuits the LLM call.

`flag_evaluated_at` is set **after** the LLM task completes, not at dispatch — a crash
before completion causes a duplicate dispatch later, which `TranscriptFlag`'s UNIQUE
constraint on `(call_id, turn_index, source, matched_phrase)` rejects harmlessly.
Marking evaluated at dispatch time would risk losing a safety flag permanently on the
same crash. Duplicate spend is recoverable; a silently dropped flag is not.

`role == "unknown"` turns are skipped (logged) from both tiers and marked evaluated
immediately — an unclassifiable turn surfaces via the log line rather than being
silently guessed into either ruleset.
